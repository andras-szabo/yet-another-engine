About DLL hot reload.

Basic idea: whenever the game DLL is recompiled, unload the currently held library
(in memory), and replace it with the new one, while keeping scene state in memory,
in as much as this is possible.

The reload cycle:
Detect change -> wait for file unlock -> SerializeScene (in-memory)
 -> DllLoader::Unload -> copy new DLL to shadow path
 -> DllLoader::Load (from shadow path) -> DeserializeScene ->
 swap active scene.


Implementation plan:

Step 1. FileWatcher:

Windows API: ReadDirectoryChangesW with overlapped I/O - no background thread
 	      needed.

From the Windows API docs:
"To obtain a handle to a directory, use the CreateFile function with the
FILE_FLAG_BACKUP_SEMANTICS flag.

"A call to ReadDirectoryChangesW can be completed synchronously or asynchronously. To
specify async completion [...], additionally specify the FILE_FLAG_OVERLAPPED attribute
in the dwFlagsAndAttributes parameter to CreateFile. (see above.) Then specify
an OVERLAPPED structure when you call ReadDirectoryChangesW.

"When you first call RDCW, the system allocates a buffer to store change info.
This buffer is associated with the dir handle until it is closed, and its size
does not change during its lifetime. Directory changes that occur between calls
to this function are added to the buffer and then returned with the next call.
If the buffer overflows, ReadDirectoryChangesW will still return true, but the
entiry contents of the buffer are discared, and the lpBytesReturned parameter
will be zero, which indicates that your buffer was too small to hold all of the
changes that occurred.

"[...] For async completion, you can receive notification in one of three ways.
* Using the GetOverlappedResult function.
	Do not specify a completion routine in lpCompletionRoutine parameter,
	and set hEvent member of the OVERLAPPED structure to a unique event.

* Using the GetQueuedCompletionStatus function.
	Do not specify a completion routine; associate the directory handle
	with a completion port by calling the CreateIoCompletionPort function.

* Using a completion routine.
	Specify a completion routine in lpCompletionRoutine. This will be called
	whenever the operation has been completed or canceled while the thread
	is in an alertable wait state."

	OK, so:

	1.1. Open a directory handle, issue the first ReadDirectoryChangesW, with
		the appropriate flags.
	1.2. Poll() gets overlapped IO result, with bWait = FALSE. Non-blocking.
		If this returns TRUE and the changed file name matches the target,
		sets a dirty flag, reissues ReadDirectoryChangesW for the next
		event, and returns true.
	1.3. Destructor closes both the directory handle and the overlapped evt
		handle.

	This avoids threading complexity, while being more precise than
	FindFirstChangeNotification (which only tells you that something has changed,
	but not what exactly).

2. Shadow copy strategy - in DllLoader or a free function

	When Windows loads a DLL via LoadLibraryW, it locks the file, so the build
	system can't overwrite it. Solution: always load from a copy.

	2.1. On Initialize: copy game-template.dll -> game-template_shadow.dll, load
		the shadow.
	2.2. On Reload: FreeLibrary releases the shadow, then copy then new original
		over it, then reload using LoadLibraryW.

	(2.3. Consider alternating pairs: _shadow_0.dll, _shadow_1.dll, to avoid any
	'file still locked' race, although a single shadow name should work fine, as
	long as FreeLibrary has completed before the copy.)

	Add CopyDllToShadow(originalPath, shadowPath) -> Expected<void> using CopyFileW.

3. "File ready" guard

	After FileWatcher::Poll() returns true, the linker may still be writing. Retry
	opening the original DLL with 
	CreateFile(..., GENERIC_READ, 0 /* exclusive share */, ...)
	in a small loop (e.g. up to 20 x 50 ms). Return an error if it never becomes
	available.

4. HotReloadManager

	In the editor project, HotReloadManager.ixx.
	This lives in the editor since it's orchestration logic, not engine infrastr.

	Initialize(dllPath, watchDirectory):
		compute shadow path
		CopyDllToShadow(dllPath, shadowPath)
		DllLoader::Load(shadowPath)
		FileWatcher::Start(watchDirectory, dllPath.filename)

	Poll(scene&, componentStorage&) -> ReloadStatus { NoChange, Succeeded, Failed }:
		if not FileWatcher::Poll() -> return NoChange
		WaitForFileReady(dllPath)
		DataFile snapshot
		SerializeScene(scene, snapshot)	// in-memory, no disk I/O
		Scene::TearDown()		// destroys all GOs
		ComponentStorage::Clear()	// clear anything left over
		DllLoader::Unload
		CopyDllToShadow(dllPath, shadowPath)
		DllLoader::Load
		auto newScene = DeserializeScene(snapshot, componentStorage)
		if ok -> EngineInstance::SetActiveScene(newScene) -> return succeeded
		else -> log error -> return failed

5. ComponentStorage handling

	This needs a decision. DeserializeScene allocates new components into an existing
	componentStorage. The old scene's components are still alive in that storage,
	creating a memory leak and potentially dangling state.

	Two options:
	A - replace ComponentStorage entirely on reload; create a fresh one, deserialize
		into it, swap it in EngineInstance. Old storage is then destroyed.
	
	B - add ComponentStorage::Clear() to destroy previous components and reuse the
		same storage.

	Claude says option A is cleaner. But I'm not sure TBH. From perf. considerations,
	I think I'd actually like to keep the exsting storage, and reuse it.

	=> Let's go with B; this means that we have to tear down the scene BEFORE calling
	FreeLibrary. Fair enough. (adjusted Poll sequence above)

		=> this means we need to:

			o Scene::TearDown(); call OnDestroy on components.
			- EngineInstance::SetActiveScene() w/ lightweight setter.

6. Main loop pre-requisite (in the editor)

	Editor needs a loop, even a trivial while(running) polling HotReloadManager::Poll().
