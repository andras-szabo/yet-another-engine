 What a Logger needs to do

  At its core, it's a simple pipeline: your code calls a log function ? the logger formats a message ? it writes
  that message to one or more sinks (console, file, etc.).

  The severity levels act as a filter — you set a minimum level (e.g. Info), and anything below it (e.g. Trace,
  Debug) is silently dropped. Useful for keeping release logs quiet.

  ----------------------------------------------------------------------------------------------------------------

  Intended functionality

   - Five levels: Trace < Debug < Info < Warn < Error
   - Configurable minimum level — anything below it is a no-op
   - Console sink — coloured output (Windows 10+ supports ANSI escape codes natively), Warn in yellow, Error in
  red, etc.
   - File sink — plain text, appended to a file (e.g. engine.log), opened once at startup
   - Formatted messages — using std::format (C++20), so you can write LOG_INFO("Loaded {} assets", count)
   - Automatic source location — using std::source_location (C++20), so every log line automatically includes the
  file name and line number without you typing it

  ----------------------------------------------------------------------------------------------------------------

  Implementation sketch

  Three pieces:

  1. A LogLevel enum and a Sink interface

   enum class LogLevel { Trace, Debug, Info, Warn, Error };

   struct Sink {
       virtual void Write(LogLevel, std::string_view message) = 0;
       virtual ~Sink() = default;
   };

  2. ConsoleSink and FileSink — both implement Sink::Write. ConsoleSink picks an ANSI colour based on level;
  FileSink writes to an std::ofstream.

  3. A Logger class — holds a vector<unique_ptr<Sink>>, a minimum level, and a Log() method that formats the
  message and fans it out to all sinks. Exposed as a global instance (not a singleton pattern, just a global in the
  module) so any code can reach it without passing it around.

  Convenience layer on top — either free functions or macros:

   LOG_INFO("Window created: {}x{}", width, height);
   LOG_WARN("Asset not found: {}", path);

  Macros are slightly preferable here because they can capture std::source_location at the call site automatically
  without the caller having to think about it.

  ----------------------------------------------------------------------------------------------------------------

  Where it lives

  Since both editor and game-template link against engine-core, the logger goes into engine-core as a new .ixx
  module interface unit — one file. Both processes get logging for free.

  ----------------------------------------------------------------------------------------------------------------

  Things to decide before implementing

   - Should Error also throw / abort? Some engines make Error fatal. Probably not for MVP — just log and let the
  caller decide.
   - Timestamp format — wall-clock time (e.g. [14:22:05.123]) vs elapsed time since startup? Elapsed is simpler and
   often more useful during development.
   - Thread safety — a simple std::mutex around the sink fan-out is enough for MVP, or we skip it entirely for now
  since nothing is multithreaded yet.
